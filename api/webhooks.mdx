Webhooks let your application receive real-time HTTP callbacks whenever specific events occur in Fluid. Instead of polling the API, you register a URL and Fluid sends a `POST` request to that URL each time a matching event fires — whether that's a cart being updated, an order placed, a payment processed, or a Droplet installed. Use webhooks to keep your own systems in sync with Fluid without continuous polling.

## Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| `GET` | `/api/company/webhooks` | List webhooks |
| `POST` | `/api/company/webhooks` | Create a webhook |
| `GET` | `/api/company/webhooks/{id}` | Get a webhook |
| `PUT` | `/api/company/webhooks/{id}` | Update a webhook |
| `DELETE` | `/api/company/webhooks/{id}` | Delete a webhook |
| `DELETE` | `/api/company/webhooks/bulk_delete` | Bulk delete webhooks |
| `GET` | `/api/company/webhook_events` | View event history |
| `POST` | `/api/company/webhook_events/{id}/retrigger` | Retry a failed event |
| `GET` | `/api/webhooks/schemas` | List available event schemas |
| `GET` | `/api/webhooks/schemas/{event_identifier}` | Get a specific event schema |

## Create a Webhook

Register a new webhook by specifying the resource and event you want to listen for, along with the destination URL Fluid should call. Your URL must use `https://`.

```bash
curl -X POST https://api.fluid.app/api/company/webhooks \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "webhook": {
      "resource": "order",
      "event": "created",
      "url": "https://my-app.com/webhooks/order-created",
      "active": true,
      "http_method": "post",
      "auth_token": "my-secret-token-123"
    }
  }'
```

**Required fields**

<ParamField body="resource" type="string" required>
  The resource type to listen on. Examples: `cart`, `order`, `payment`, `droplet`, `subscription`. Retrieve valid values from `GET /api/webhooks/schemas`.
</ParamField>

<ParamField body="event" type="string" required>
  The event name within that resource. Examples: `created`, `updated`, `installed`, `uninstalled`. Must be a valid event for the chosen resource.
</ParamField>

<ParamField body="url" type="string" required>
  The HTTPS destination URL Fluid will send webhook payloads to. Plain `http://` URLs are not accepted.
</ParamField>

<ParamField body="active" type="boolean">
  Set to `true` to enable the webhook immediately. Defaults to `true`.
</ParamField>

<ParamField body="http_method" type="string">
  The HTTP method Fluid uses when calling your URL. One of `post`, `put`, or `get`. Defaults to `post`.
</ParamField>

<ParamField body="auth_token" type="string">
  An optional token included in the `Authorization` header of outgoing webhook requests, allowing your endpoint to verify the request came from Fluid.
</ParamField>

**Response**

```json
{
  "webhook": {
    "id": 42,
    "resource": "order",
    "event": "created",
    "event_identifier": "order_created",
    "url": "https://my-app.com/webhooks/order-created",
    "active": true,
    "http_method": "post",
    "auth_token": "my-secret-token-123"
  },
  "meta": {
    "request_id": "req_01HXYZ",
    "timestamp": "2025-06-01T12:00:00Z"
  }
}
```

---

## Webhook Object

<ResponseField name="id" type="integer">
  Unique numeric identifier for the webhook.
</ResponseField>

<ResponseField name="url" type="string">
  The destination URL Fluid sends payloads to. Always `https://`.
</ResponseField>

<ResponseField name="active" type="boolean">
  Whether the webhook is currently active and receiving events.
</ResponseField>

<ResponseField name="resource" type="string">
  The resource type this webhook is scoped to (e.g. `cart`, `order`, `droplet`).
</ResponseField>

<ResponseField name="event" type="string">
  The event name within the resource (e.g. `updated`, `installed`).
</ResponseField>

<ResponseField name="event_identifier" type="string">
  The combined identifier for the event, formed as `{resource}_{event}` (e.g. `cart_updated`, `order_created`). Use this to look up the payload schema.
</ResponseField>

<ResponseField name="http_method" type="string">
  The HTTP method used for delivery: `post`, `put`, or `get`.
</ResponseField>

<ResponseField name="auth_token" type="string or null">
  Optional authentication token sent in the `Authorization` header of outbound webhook requests. Returns `null` if not set.
</ResponseField>

## List Webhooks

Retrieve all registered webhooks for your company.

```bash
curl https://api.fluid.app/api/company/webhooks \
  -H "Authorization: Bearer <token>"
```

---

## Update a Webhook

Update an existing webhook's URL, event, active state, or any other field using `PUT`.

```bash
curl -X PUT https://api.fluid.app/api/company/webhooks/42 \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "webhook": {
      "url": "https://my-app.com/webhooks/order-created-v2",
      "active": true
    }
  }'
```

---

## Delete Webhooks

Delete a single webhook by ID, or bulk-delete multiple webhooks in one request.

```bash
# Delete a single webhook
curl -X DELETE https://api.fluid.app/api/company/webhooks/42 \
  -H "Authorization: Bearer <token>"

# Bulk delete
curl -X DELETE https://api.fluid.app/api/company/webhooks/bulk_delete \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{ "ids": [42, 43, 44] }'
```

## Querying Webhook History

Use the webhook events endpoint to inspect your delivery history, diagnose failures, and review payloads. You can filter by date, event type, or delivery status.

```bash
curl "https://api.fluid.app/api/company/webhook_events?event_identifier=order_created&status=failed&from=2025-05-01&to=2025-05-31" \
  -H "Authorization: Bearer <token>"
```

<ParamField query="event_identifier" type="string">
  Filter events by identifier (e.g. `order_created`, `cart_updated`).
</ParamField>

<ParamField query="status" type="string">
  Filter by delivery status. Values: `delivered`, `failed`, `pending`.
</ParamField>

<ParamField query="from" type="string">
  Start of the date range in `YYYY-MM-DD` format.
</ParamField>

<ParamField query="to" type="string">
  End of the date range in `YYYY-MM-DD` format.
</ParamField>

## Retrying Failed Events

If a webhook delivery fails (for example, your endpoint was temporarily unavailable), you can retrigger any past event by its ID. Fluid will re-send the original payload to the registered URL.

```bash
curl -X POST https://api.fluid.app/api/company/webhook_events/evt_01HXYZ/retrigger \
  -H "Authorization: Bearer <token>"
```

<Note>
  Retriggers are rate-limited. You can retrigger a given event up to 10 times. If your endpoint continues to fail, check that it returns a `2xx` status code within 10 seconds.
</Note>

## Available Event Schemas

Retrieve a list of all supported event types and their payload schemas. Use this to discover valid `resource` and `event` combinations when registering webhooks.

```bash
# List all available event schemas
curl https://api.fluid.app/api/webhooks/schemas \
  -H "Authorization: Bearer <token>"

# Get the schema for a specific event
curl https://api.fluid.app/api/webhooks/schemas/order_created \
  -H "Authorization: Bearer <token>"
```

The schema response describes the exact payload structure Fluid sends to your endpoint when the event fires, including field names, types, and example values.
