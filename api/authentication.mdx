Company endpoints on the Fluid API are authenticated with a bearer token. Fluid provides three token types for integrators — company, partner, and public — each designed for a different trust level and integration pattern. Public storefront read endpoints require no token at all; everything below applies to the token-authenticated company surface.

## Passing your token

Include your token in the `Authorization` header, prefixed with `Bearer `:

```bash
curl https://api.fluid.app/api/v202604/company/products \
  -H "Authorization: Bearer <your_token>" \
  -H "Content-Type: application/json"
```

A request with a missing, invalid, expired, or revoked token receives a `401 Unauthorized`. The `401` body is a bare `{ "message": "..." }` object rather than the wrapped error envelope used by `404` and `422` responses — see [Errors](/api/overview#errors) for the full envelope contract.

## Token types

Each token type has a different scope of access and a different lifecycle.

### Company token

A company token grants full administrative access to your Fluid account, equivalent to a logged-in admin. Use company tokens for server-side integrations, internal tooling, and automation scripts where you fully control the runtime environment.

Create a company token from your Fluid dashboard under **Settings → Developer**. This is the primary way to obtain a company token.

<Warning>
  Never use a company token in client-side JavaScript or commit it to a public repository. Anyone who obtains a company token has full access to your account.
</Warning>

### Partner token

A partner token (prefixed `PT-`) provides company-level access for third-party integrations. Partner tokens support an **optional** expiration and can be tied to a role, making them safer to distribute than a raw company token. They suit agency-built integrations, certified app partners, and automated workflows that need elevated access without sharing your primary company token.

Partner tokens are managed under `/api/v2025-06/tokens/partner` and require a company-admin token to create or manage.

### Public token

A public token (prefixed `pub-`) is designed for client-side use and is scoped to a specific set of read/asset permissions, so it can be embedded in browser code or mobile apps. Public tokens are managed under `/api/v2025-06/tokens/public` and, like partner tokens, require a company-admin token to create or manage.

Every public token must be constrained by an expiration, a domain allowlist, or both:

- **Short-lived** — set `expires_at`. The token stops working after that time, limiting exposure if it leaks.
- **Long-lived (non-expiring)** — omit `expires_at` and provide a `domain_allowlist`. Requests are only accepted from the listed origins.

<Warning>
  A **non-expiring** public token requires a `domain_allowlist` — creating one without either `expires_at` or a non-empty `domain_allowlist` fails validation. For a short-lived token a domain allowlist is optional but recommended.
</Warning>

## Managing partner tokens

Partner tokens live under `/api/v2025-06/tokens/partner` and support the full lifecycle: list (`GET`), show (`GET /{id}`), create (`POST`), update (`PATCH /{id}`), and delete (`DELETE /{id}`). Create one with a company-admin token by posting a `label` (required) and an optional `expires_at`:

```bash
curl -X POST https://api.fluid.app/api/v2025-06/tokens/partner \
  -H "Authorization: Bearer <your_company_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "partner_token": {
      "label": "Acme Analytics Partner",
      "expires_at": "2026-12-31T00:00:00Z"
    }
  }'
```

Omit `expires_at` for a non-expiring partner token. To revoke a partner token, send `DELETE /api/v2025-06/tokens/partner/{id}`.

## Managing public tokens

Public tokens live under `/api/v2025-06/tokens/public` and support list (`GET`), create (`POST`), and delete (`DELETE /{id}`). Create one with a company-admin token; the request body nests under `public_token` and a `name` is required.

**Create a short-lived public token:**

```bash
curl -X POST https://api.fluid.app/api/v2025-06/tokens/public \
  -H "Authorization: Bearer <your_company_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "public_token": {
      "name": "Storefront catalog widget",
      "scopes": ["products:read", "media:read"],
      "expires_at": "2026-12-31T00:00:00Z"
    }
  }'
```

**Create a non-expiring public token with a domain allowlist:**

```bash
curl -X POST https://api.fluid.app/api/v2025-06/tokens/public \
  -H "Authorization: Bearer <your_company_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "public_token": {
      "name": "Acme storefront",
      "scopes": ["products:read", "dam:browse"],
      "domain_allowlist": ["https://shop.acmewellness.com", "https://www.acmewellness.com"]
    }
  }'
```

To list your public tokens, send `GET /api/v2025-06/tokens/public`. To revoke one, send `DELETE /api/v2025-06/tokens/public/{id}`.

### Public token scopes

Request only the scopes your client-side code actually needs.

| Scope | Access |
|-------|--------|
| `dam:upload` | Upload files to the DAM |
| `dam:browse` | Browse and search DAM assets |
| `dam:unsplash` | Search and import images from Unsplash via the DAM |
| `dam:ai_generate` | Generate images using AI tools in the DAM |
| `media:read` | Read media records |
| `products:read` | Read product catalog data |
| `enrollments:read` | Read enrollment records |
| `playlists:read` | Read playlist definitions |
| `forms:read` | Read form definitions |

## Storefront company permissions

The v2026-04 storefront company endpoints (`/api/v202604/company/...`) accept a company-admin bearer token carrying `storefront.view` for reads and `storefront.update` for writes. Legacy per-resource grants (such as `products.view`) are still accepted during the migration window.

## Best practices

- **Store tokens in environment variables.** Never hard-code tokens in source files. Use `.env` files locally and a secrets manager (such as AWS Secrets Manager or Vault) in production.
- **Use the least-privileged token type.** Prefer public tokens for client-side code, partner tokens for third-party integrations, and reserve company tokens for fully trusted server-side environments.
- **Scope public tokens narrowly.** Request only the scopes a client needs, and constrain each token with an expiration, a domain allowlist, or both.
- **Prefer expiring tokens.** Even for long-running integrations, favor tokens that expire and are refreshed over indefinite ones.
- **Rotate and audit.** Periodically regenerate tokens, and review and revoke unused tokens under **Settings → Developer**.
