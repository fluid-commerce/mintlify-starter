Every request to the Fluid API requires a valid bearer token. Fluid provides three token types — company, partner, and public — each designed for a different trust level and integration pattern. Choose the right token type for your use case to keep your storefront secure and your integrations scoped appropriately.

## Passing Your Token

Include your token in the `Authorization` header of every API request. The value must be prefixed with `Bearer ` followed by your token string.

```bash
curl https://api.fluid.app/api/company/v1/products \
  -H "Authorization: Bearer <your_token>" \
  -H "Content-Type: application/json"
```

Requests made without a valid token, or with an expired or revoked token, will receive a `401 Unauthorized` response.

## Token Types

Fluid supports three token types. Each has a different scope of access and a different lifecycle.

### Company Token

A company token grants full administrative access to your Fluid account, equivalent to a logged-in admin. Use company tokens for server-side integrations, internal tooling, and automation scripts where you fully control the runtime environment.

Generate a company token from your developer settings at [https://www.fluid.app/settings/developer](https://www.fluid.app/settings/developer).

<Warning>
  Never use company tokens in client-side JavaScript or expose them in public repositories. Anyone who obtains a company token has full access to your account.
</Warning>

### Partner Token

A partner token provides company-level access for third-party integrations. Partner tokens support optional expiration, making them safer to distribute to external systems. Create a partner token by calling the partner token endpoint with your company token.

Partner tokens are ideal for agency-built integrations, certified app partners, and automated workflows that need elevated access without sharing your primary company token.

### Public Token

A public token is designed for client-side use and always begins with the prefix `pub-`. Public tokens are scoped to specific permissions and can be safely embedded in browser code or mobile apps.

Public tokens operate in one of two modes:

- **Short-lived**: Requires an `expires_at` timestamp. The token becomes invalid after that time, limiting exposure if it is intercepted.
- **Long-lived**: Requires a `domain_allowlist` array. Requests are only accepted from the specified origins, preventing use from unauthorized domains.

<Warning>
  A public token without either `expires_at` or `domain_allowlist` will fail validation and cannot be created. You must provide one or both fields.
</Warning>

## Creating a Partner Token

Send a `GET` request to the partner token endpoint using your company token. The response contains a new partner token you can distribute to your integration.

```bash
curl https://api.fluid.app/api/v2025-06/tokens/partner \
  -H "Authorization: Bearer <your_company_token>" \
  -H "Content-Type: application/json"
```

To create a partner token with an expiration date, include an `expires_at` field in the request body:

```bash
curl -X POST https://api.fluid.app/api/v2025-06/tokens/partner \
  -H "Authorization: Bearer <your_company_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "expires_at": "2025-12-31T23:59:59Z"
  }'
```

## Public Tokens

Public tokens give you fine-grained control over what client-side code can access. Create a public token by sending a `POST` request to `/api/v2025-06/tokens/public`.

**Create a short-lived public token:**

```bash
curl -X POST https://api.fluid.app/api/v2025-06/tokens/public \
  -H "Authorization: Bearer <your_company_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "scopes": ["products:read", "media:read"],
    "expires_at": "2025-06-30T00:00:00Z"
  }'
```

**Create a long-lived public token with a domain allowlist:**

```bash
curl -X POST https://api.fluid.app/api/v2025-06/tokens/public \
  -H "Authorization: Bearer <your_company_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "scopes": ["products:read", "dam:browse"],
    "domain_allowlist": ["https://www.yourstore.com", "https://app.yourstore.com"]
  }'
```

To list all public tokens for your account, send a `GET` request to `/api/v2025-06/tokens/public`. To revoke a token, send a `DELETE` request to `/api/v2025-06/tokens/public/{id}`.

### Public Token Scopes

Use scopes to limit what a public token can access. Request only the scopes your client-side code actually needs.

| Scope | Access |
|-------|--------|
| `dam:browse` | Browse and search DAM assets |
| `dam:upload` | Upload files to the DAM |
| `media:read` | Read media records |
| `products:read` | Read product catalog data |
| `enrollments:read` | Read enrollment records |
| `forms:read` | Read form definitions |

## Best Practices

Following these practices will help you keep your API credentials secure across all environments.

- **Store tokens in environment variables.** Never hard-code tokens in source files. Use `.env` files locally and secret management tools (such as AWS Secrets Manager or Vault) in production.
- **Rotate tokens regularly.** Periodically expire and regenerate tokens, especially for long-running integrations.
- **Use the least-privileged token type.** Prefer public tokens for client-side code, partner tokens for third-party integrations, and reserve company tokens for fully trusted server-side environments.
- **Set expiration on partner tokens.** Even for long-running integrations, prefer tokens that expire and are refreshed automatically over indefinite tokens.
- **Audit token usage.** Review active tokens periodically at [https://www.fluid.app/settings/developer](https://www.fluid.app/settings/developer) and revoke any that are no longer in use.
