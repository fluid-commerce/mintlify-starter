Fluid uses bearer tokens to authenticate every API request. There are three token types — company tokens for direct admin access, partner tokens for third-party integrations, and public tokens for browser and SDK use. Choose the right token type for your use case to keep your integration secure and appropriately scoped.

## Company Tokens

Company tokens grant full admin access to your Fluid account. They are the most powerful credential type and are intended for secure server-side use only.

To generate a company token, navigate to **Admin Settings → Developer** in your Fluid dashboard at [fluid.app/settings/developer](https://www.fluid.app/settings/developer). Copy the token and store it immediately — it will not be shown again.

Pass your token in the `Authorization` header on every request:

```bash
curl https://api.fluid.app/api/company/v1/products \
  -H "Authorization: Bearer <your_company_token>"
```

<Warning>
  Never use company tokens in client-side JavaScript or expose them in a public repository. Anyone who obtains your company token has full admin access to your Fluid account.
</Warning>

## Partner Tokens

Partner tokens grant company-level access to Fluid on behalf of a third-party integration. They support expiration dates, making them well-suited for time-limited or delegated access scenarios.

Create a partner token by calling `POST /api/v2025-06/tokens/partner`:

```bash
curl -X POST https://api.fluid.app/api/v2025-06/tokens/partner \
  -H "Authorization: Bearer <your_company_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "expires_at": "2025-12-31T23:59:59Z"
  }'
```

The response includes your new partner token. Use it in the `Authorization` header exactly as you would a company token:

```bash
curl https://api.fluid.app/api/v2025-06/tokens/partner \
  -H "Authorization: Bearer <your_partner_token>"
```

Retrieve an existing partner token at any time with:

```bash
curl https://api.fluid.app/api/v2025-06/tokens/partner \
  -H "Authorization: Bearer <your_company_token>"
```

## Public Tokens

Public tokens (prefixed with `pub-`) are designed for use in browsers and the FairShare client-side SDK. Because they are exposed to end users, they are scoped to specific capabilities and can be restricted by domain or expiration.

There are two modes:

- **Short-lived tokens** — include an `expires_at` timestamp. No domain allowlist is required. Use these for temporary sessions or testing.
- **Long-lived tokens** — omit `expires_at` and include a `domain_allowlist` instead. Fluid enforces the allowlist so only requests from approved origins are accepted (for example, `*.mystore.com`).

<Warning>
  A public token without either `expires_at` or a `domain_allowlist` will fail validation. Every public token must have at least one of these set.
</Warning>

### Create a Public Token

```bash
# Short-lived token (expires in 24 hours)
curl -X POST https://api.fluid.app/api/v2025-06/tokens/public \
  -H "Authorization: Bearer <your_company_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "scopes": ["products:read", "media:read"],
    "expires_at": "2025-06-15T12:00:00Z"
  }'

# Long-lived token (domain-restricted)
curl -X POST https://api.fluid.app/api/v2025-06/tokens/public \
  -H "Authorization: Bearer <your_company_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "scopes": ["products:read", "dam:browse"],
    "domain_allowlist": ["*.mystore.com", "mystore.com"]
  }'
```

### List and Delete Public Tokens

```bash
# List all public tokens
curl https://api.fluid.app/api/v2025-06/tokens/public \
  -H "Authorization: Bearer <your_company_token>"

# Delete a public token
curl -X DELETE https://api.fluid.app/api/v2025-06/tokens/public/<token_id> \
  -H "Authorization: Bearer <your_company_token>"
```

### Public Token Scopes

Choose only the scopes your integration actually needs. Applying the principle of least privilege reduces the impact of a token being leaked.

| Scope | Description |
|---|---|
| `dam:browse` | Browse assets in the Digital Asset Manager |
| `dam:upload` | Upload assets to the Digital Asset Manager |
| `dam:unsplash` | Search and import images from Unsplash |
| `dam:ai_generate` | Generate images using AI tools |
| `media:read` | Read media resources |
| `products:read` | Read product catalog data |
| `enrollments:read` | Read enrollment records |
| `playlists:read` | Read playlist data |
| `forms:read` | Read form definitions and submissions |

## Best Practices

- **Store tokens in environment variables.** Never hard-code tokens in source files. Use a secrets manager or `.env` file that is excluded from version control.
- **Never expose company tokens client-side.** Use public tokens with the minimum required scopes for all browser-facing integrations.
- **Set expiration on short-lived tokens.** When issuing tokens for temporary sessions, always provide an `expires_at` value to limit the window of exposure.
- **Rotate tokens regularly.** Periodically generate new tokens and revoke old ones, especially after team member changes or a suspected compromise.
- **Use domain allowlists for long-lived public tokens.** Restricting origins to your own domains prevents tokens from being reused on unauthorized sites.
