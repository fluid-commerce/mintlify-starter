Fluid Orchestration is a routing engine that distributes card transactions across multiple payment service providers (PSPs). Instead of relying on a single processor, you can split traffic by percentage, route by transaction attributes like country or payment method, and define a fallback chain for when no specific rule applies. This reduces dependency on any one PSP and lets you optimize for cost, approval rates, or regional coverage.

## How Routing Works

When a transaction arrives, Fluid Orchestration evaluates your active configuration in the following order:

1. **Active configuration check** — If a configuration is active, Fluid applies it. Only one configuration can be active at a time.
2. **Volume-based or rule-based routing** — The active configuration routes the transaction either by percentage distribution or by matching rule conditions.
3. **Default fallback** — If no active configuration exists, or no rule matches, Fluid routes through your priority-ordered fallback list.

## Volume-Based Routing

Volume-based routing splits traffic across processors by percentage. This is useful for gradual rollouts, A/B testing PSPs, or maintaining redundancy. The `distributions` array must contain percentages that sum exactly to `100`.

```bash
curl -X POST https://api.fluid.app/api/fluid_orchestration/configurations \
  -H "Authorization: Bearer <your_company_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "configuration_type": "volume_based",
    "name": "70-30 Split",
    "distributions": [
      {
        "processor": "stripe",
        "percentage": 70
      },
      {
        "processor": "braintree",
        "percentage": 30
      }
    ]
  }'
```

<Tip>
  When adding a new processor for the first time, start with a conservative split such as 90/10. Monitor approval rates and error logs before increasing the share sent to the new processor.
</Tip>

## Rule-Based Routing

Rule-based routing lets you define conditions so that specific transaction types are always directed to a particular processor. Each rule in the `rules` array is evaluated in order — the first match wins.

The example below routes Apple Pay transactions to Stripe and US transactions over $500 to Braintree:

```bash
curl -X POST https://api.fluid.app/api/fluid_orchestration/configurations \
  -H "Authorization: Bearer <your_company_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "configuration_type": "rule_based",
    "name": "Apple Pay and High-Value US Routing",
    "rules": [
      {
        "processor": "stripe",
        "conditions": {
          "payment_source": "apple_pay"
        }
      },
      {
        "processor": "braintree",
        "conditions": {
          "country_iso": "US",
          "amount_greater_than": 500
        }
      }
    ]
  }'
```

### Supported Rule Conditions

Combine one or more conditions within a single rule. All specified conditions must match for the rule to apply.

| Condition | Type | Description |
|---|---|---|
| `payment_source` | string | Payment method type (e.g. `apple_pay`, `google_pay`, `card`) |
| `country_iso` | string | Two-letter ISO 3166-1 country code of the billing address |
| `amount_greater_than` | number | Matches transactions above this amount (in the transaction's currency) |
| `amount_less_than` | number | Matches transactions below this amount (in the transaction's currency) |
| `currency_code` | string | Three-letter ISO 4217 currency code (e.g. `USD`, `EUR`, `GBP`) |

## Default Fallback

The default fallback is a priority-ordered list of processors that Fluid uses when no active configuration exists or no rule condition is matched. The first processor in the list is tried first.

Retrieve your current fallback configuration:

```bash
curl https://api.fluid.app/api/fluid_orchestration/default_fallback \
  -H "Authorization: Bearer <your_company_token>"
```

Update the fallback priority order:

```bash
curl -X PATCH https://api.fluid.app/api/fluid_orchestration/default_fallback \
  -H "Authorization: Bearer <your_company_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "processors": ["stripe", "braintree", "square"]
  }'
```

## Activating a Configuration

After creating a configuration, activate it to make it live. This replaces any previously active configuration.

```bash
curl -X POST https://api.fluid.app/api/fluid_orchestration/configurations/<config_id>/activate \
  -H "Authorization: Bearer <your_company_token>"
```

To deactivate a configuration without activating another:

```bash
curl -X POST https://api.fluid.app/api/fluid_orchestration/configurations/<config_id>/deactivate \
  -H "Authorization: Bearer <your_company_token>"
```

Update an existing configuration with `PATCH`:

```bash
curl -X PATCH https://api.fluid.app/api/fluid_orchestration/configurations/<config_id> \
  -H "Authorization: Bearer <your_company_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Updated Split"
  }'
```

<Warning>
  Only one configuration can be active at a time. You must deactivate a configuration before you can delete it. Attempting to delete an active configuration will return an error.
</Warning>

To delete a deactivated configuration:

```bash
curl -X DELETE https://api.fluid.app/api/fluid_orchestration/configurations/<config_id> \
  -H "Authorization: Bearer <your_company_token>"
```

## Check Orchestration Status

Verify the current status of the Fluid Orchestration engine at any time:

```bash
curl https://api.fluid.app/api/fluid_orchestration/status \
  -H "Authorization: Bearer <your_company_token>"
```
